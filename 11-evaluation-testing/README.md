# Module 3 · 评测与测试

目标：建立可重复的质量评估体系，避免靠感觉判断大模型应用是否变好。

## 为什么需要评测

LLM 应用的输出不稳定。一次看起来正确，不代表系统可靠。每次改 Prompt、模型、chunk、检索参数或工具逻辑，都可能引入回归。

## 评测对象

### RAG 评测

- 是否召回正确文档。
- 答案是否基于上下文。
- 答案是否完整。
- 是否拒答无依据问题。
- 引用是否正确。

### Agent 评测

- 是否选择正确工具。
- 参数是否正确。
- 是否按正确顺序执行。
- 是否避免危险操作。
- 失败时是否能恢复。

### API 评测

- 接口是否返回正确状态码。
- 错误输入是否被校验。
- 超时和异常是否可控。
- 响应格式是否稳定。

## 评测集格式

建议用 JSONL：

```json
{"id":"q1","question":"退货期限是多久？","expected_keywords":["7天","退货"],"expected_source":"sample.txt"}
{"id":"q2","question":"可以线下门店退货吗？","expected_answer_type":"not_found"}
```

## 自动评测维度

- `retrieval_hit`：是否召回目标文档。
- `answer_contains_keywords`：是否包含关键事实。
- `grounded`：是否没有脱离上下文编造。
- `format_valid`：JSON 或结构化输出是否有效。
- `tool_sequence_match`：Agent 工具调用顺序是否符合预期。

## 推荐练习

1. 为知识库问答创建 20 条 JSONL 评测样例。
2. 写一个脚本批量请求 `/qa` 接口并输出得分。
3. 修改 chunk size，比较召回命中率变化。
4. 为 Agent 写 10 条工具调用评测样例。
5. 把评测脚本加入 CI 或本地提交前检查。

## 验收标准

完成本模块后，你应该能实现：

- 每次改动后自动跑评测集。
- 输出 RAG 召回率和答案命中率。
- 保存失败案例，方便定位问题。
- 用数据比较不同模型、Prompt、检索参数的效果。
