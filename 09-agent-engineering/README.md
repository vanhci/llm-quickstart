# Module 1 · Agent 工程化

目标：把 Day 3 的简单工具调用升级为可维护、可调试、可扩展的 Agent 系统。

## 为什么需要工程化 Agent

简单 Agent Demo 通常只演示“模型决定调用工具”。真实业务里还需要处理：

- 工具参数错误。
- 工具调用超时。
- 模型陷入循环。
- 用户请求涉及高风险操作。
- 多步骤任务中间状态丢失。
- 不同工具需要不同权限。
- 执行结果需要被记录、回放和评估。

工程化 Agent 的核心不是让模型“更自由”，而是把模型放进受控执行框架里。

## 推荐架构

```text
用户请求
  ↓
意图识别与风险判断
  ↓
任务状态 State
  ↓
规划器 Planner
  ↓
工具选择 Tool Router
  ↓
工具执行 Tool Executor
  ↓
观察结果 Observation
  ↓
是否继续 / 是否需要人工确认
  ↓
最终回答
```

## 必须掌握的能力

### 1. 工具注册

每个工具都应该有清晰的 schema：

- 工具名。
- 参数 JSON Schema。
- 返回值结构。
- 超时时间。
- 是否需要人工确认。
- 权限等级。

### 2. 状态管理

Agent 不应该只靠一串 `messages` 工作。至少要记录：

- 当前任务目标。
- 已执行步骤。
- 每步工具输入和输出。
- 错误信息。
- 用户确认记录。
- 最大执行步数。

### 3. 失败处理

需要明确区分：

- 参数错误：让模型修正参数。
- 外部服务失败：重试或降级。
- 权限不足：拒绝或请求用户授权。
- 任务不可完成：给出原因和替代方案。

### 4. 人工确认

以下操作必须确认：

- 发邮件、发消息。
- 写数据库。
- 删除文件。
- 下单、支付、转账。
- 调用外部系统产生不可逆影响。

### 5. 可观测性

记录每次 Agent 执行轨迹：

- 输入。
- 模型输出。
- 工具调用。
- 工具结果。
- 耗时。
- token 用量。
- 最终答案。

## 推荐练习

1. 把 `03-agent-tool-use/examples/01_tool_calling.py` 改成工具注册表。
2. 给每个工具增加 `timeout`、`risk_level`、`requires_confirmation`。
3. 实现一个 `AgentState`，记录完整执行轨迹。
4. 给 `send_email` 工具加人工确认逻辑。
5. 给 Agent 增加最大步数和重复工具调用检测。

## 验收标准

完成本模块后，你应该能实现：

- 一个包含 3 个以上工具的 Agent。
- 每个工具都有 schema 和安全等级。
- Agent 能在工具失败后重试或优雅失败。
- 高风险工具会先请求确认。
- 每次执行都有可查看的 trace。
